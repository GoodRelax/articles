## なぜ作ったか

Claude Code でソフトを作っていて気づいた。

AIコーディングは外注と同じプロセスだ。仕様 → 開発 → レビュー → 検収。作り手が海外チームだろうが自分のマシン上のLLMだろうが、流れも失敗パターンも同じ。

外注ソフトの品質は3つで決まる：

1. 外注先の能力
2. 開発プロセス
3. 仕様書

1番？ Claude Code。文句なし。

2番？ 取り組み中。

3番が問題。曖昧すぎ → AIが頼んでない機能をハルシネーションで生やす。冗長すぎ → コンテキストに溺れて大事なことを見失う。

IEEE 29148 の仕様書は厳密だけど、200ページをLLMに食わせたら迷子になる。カジュアルに「Todoアプリ作って」なら通る — 認証やステートマシンが出てきた瞬間に崩壊する。

色々試して分かったこと：**従来の仕様書テンプレはAI向けに設計されてない。**

じゃあAIは仕様書に何を求めてるんだ？ その問いから作ったのが…

## ANMS — AI-Native Minimal Spec

既存の記法 — EARS、Gherkin、Mermaid — をAI駆動開発向けに再構成した仕様書テンプレート。

各記法は先人たちの偉業。自分がやったのは：ほぼ全自動開発を数ヶ月やる中でAIが迷うポイントを洗い出し、各層に最適な記法を割り当て、Clean Architectureの考え方で章構成を組んだこと。

## Stable Top, Flexible Bottom（上剛下柔）

仕様書のすべてが同じ頻度で変わるわけじゃない。

- プロジェクトの目標・制約 → めったに変わらない
- Gherkinシナリオ → しょっちゅう変わる

なのになぜ同じ重みで扱うのか？

Robert C. Martin の **安定依存の原則**（SDP）— 安定したものに依存せよ、不安定なものには依存するな — をコードではなくドキュメント構造に適用した。

```
Ch1  Foundation        ← 剛: めったに変わらない
Ch2  Requirements
Ch3  Architecture
Ch4  Specification     ← 柔: よく変わる
Ch5  Test Strategy
Ch6  Design Principles ← AIのコードレビュー基準になる
```

上の章が下の章を制約する。逆はない。

Ch4のGherkinシナリオを変えても → Ch1, Ch2は無傷。
Ch1のGoalを変えたら → それより下は全部見直し。

SDPを*ドキュメント*に適用する、コードじゃなく — これがポイント。AIに**どのコンテキストを優先すべきか**を構造的に伝える。

## 各層に最適な記法

一つの記法で全部はカバーできない。各章にフィットする記法を選んだ — 巨人の肩に乗せてもらった。

| 章 | 記法 | なぜ |
| --- | --- | --- |
| **Foundation** | 自然言語 + テーブル | 人間が目標・範囲・制約を定義 |
| **Requirements** | EARS構文 | 構造化パターンで曖昧さ排除 |
| **Architecture** | Mermaid（色分け） | 人間とAIで構造を視覚的に同期 |
| **Specification** | Gherkin | AIがテストコードを直接生成 |

### Ch1: Foundation — 自然言語 + テーブル

Goal（何を作るか）、Scope（どこまで）、Constraint（何を守るか）。平易な言葉とテーブル。

ここに一番時間をかける — その価値はある。基盤は自分の意図の結晶。

### Ch2: Requirements — EARS

「システムはエラーを適切にハンドリングすべき」

…*適切*って何だ？ AIに渡したら好き勝手にやる。

EARS（Mavin et al., 2009）で解決：

- **When** 前方障害物との衝突まで1秒以内と判定された場合, **the System shall** 即座に緊急ブレーキを作動。
- **While** 自動運転モードで走行中, **the System shall** 車線中央を±15cm以内で維持。

6パターン。曖昧さゼロ。元は組込み系 — AI駆動開発にぴったりだった。

### Ch3: Architecture — Mermaid

AI駆動開発において、Mermaid図は**イラストじゃない — 設計そのもの。**

AIはコンポーネント図を読んでファイル分割・importパス・依存方向を決める。ANMSではアーキテクチャレイヤーごとの色分けを必須にしている — Mermaidのレイアウトエンジンは気まぐれなので、色がないとどのボックスがどのレイヤーか分からない。

### Ch4: Specification — Gherkin

Gherkinシナリオは受入テストであり、TDDの文脈では実装仕様でもある。各シナリオは `(traces: FR-xxx)` で要求にトレースバック、抜け漏れなし。

## 具体例:「お抱え運転手付きの車」

全文は載せられないが、コンセプト→仕様の流れはこう。

（Foundation → EARS → Mermaidシーケンス図 → Gherkin の4ステップ例。省略）

4ステップ。コンセプトからテスト可能な仕様へ。AIは何を作り、何をテストし、何の制約を守るか分かる。

## 人間がやること

ほぼ全自動でも、3つは人間の仕事：

1. **コンセプトを決める**
2. **重要な判断を下す**
3. **成果物を確認する**

それ以外？ AIに任せて、大事なところだけレビュー。
